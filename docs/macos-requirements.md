# DriveCk macOS 需求文档

## 1. 背景与目标

DriveCk 已有共享 Rust 校验引擎、跨平台 CLI、Linux GTK 前端与 Win32 前端，但缺少一套适配 macOS 使用习惯的原生入口。本次需要补齐一套 **macOS 专用 CLI 与 GUI**，并通过 FFI 复用 Rust 核心能力，形成一致的结果模型、报告格式和校验语义。

本次目标不是增加更多校验算法，而是在 **macOS 用户体验、构建结构、平台整合、交互细节** 上补齐完整产品面。

## 2. 产品目标

1. 提供一套 **macOS 原生 CLI**，可在终端中完成设备发现、设备校验、进度输出与报告导出。
2. 提供一套 **macOS 原生 GUI**，使用 SwiftUI 为主、必要处结合 AppKit，符合现代 macOS 桌面体验。
3. 在 macOS 层完成设备发现与交互编排，在 Rust 共享层完成校验与报告生成，边界清晰。
4. 对风险操作提供清晰的安全提示，避免用户误选系统盘或已挂载设备。
5. 结果表达要足够直观，既能快速判断“是否可靠”，也能查看详细证据。

## 3. 用户与场景

### 3.1 目标用户

- 需要检测 U 盘、移动硬盘、SD 卡真实性和可用容量的普通用户
- 需要批量验盘或生成结果报告的技术用户
- 更偏好图形界面的桌面用户
- 更偏好自动化和脚本化的终端用户

### 3.2 核心场景

1. 用户插入移动存储设备后，希望快速确认设备是否为真实容量。
2. 用户在购买新存储设备后，希望做一次完整抽样校验并保存报告。
3. 用户发现设备读写异常，希望通过图形界面查看失败区域与统计指标。
4. 技术用户希望通过 CLI 在终端中执行校验，并将报告保存为文本文件。

## 4. 范围

### 4.1 本次范围内

- macOS 专用代码目录
- macOS 原生 CLI
- macOS 原生 GUI
- Swift 与 Rust 的 FFI 接入
- macOS 设备发现
- 校验任务的进度展示、取消、结果展示、报告导出
- README 中的 macOS 说明
- 需求文档与设计文档

### 4.2 本次范围外

- 新的底层校验算法
- 安装器、自动更新、签名分发
- 权限提升辅助守护进程或特权 helper
- 多语言国际化
- 历史任务数据库或云同步
- 自动卸载设备、自动修复设备

## 5. 功能需求

## 5.1 设备发现

### 5.1.1 发现范围

系统应发现适合校验的 **外接、可移除、整盘级别** 存储目标，包括但不限于：

- U 盘
- 外置 SSD / HDD
- SD 卡与读卡器设备
- 其它外接物理存储设备

### 5.1.2 过滤规则

系统默认仅展示满足以下条件的目标：

- 是整盘目标，而不是单个分区
- 不是系统内置盘
- 可以获取设备路径、容量和基础元数据

系统必须显式标记以下状态：

- 已挂载，不可直接校验
- 可移除
- USB / Thunderbolt / SD 等传输类型
- 权限不足导致无法访问

### 5.1.3 展示字段

每个候选设备至少应提供：

- 设备名
- 设备路径
- 容量
- 传输类型
- 可移除状态
- 是否已挂载
- 适合展示的型号或介质名

## 5.2 校验流程

### 5.2.1 CLI

CLI 至少支持：

- 列出可校验设备
- 指定设备执行校验
- 显示校验进度
- 指定固定随机种子
- 保存文本报告
- 在危险操作前给出确认提示

CLI 在非交互模式下，若目标为危险设备且未显式确认，必须拒绝继续。

### 5.2.2 GUI

GUI 至少支持：

1. 刷新设备列表
2. 选择设备
3. 查看设备摘要与风险状态
4. 启动校验
5. 实时查看阶段与进度
6. 主动取消校验
7. 查看最终结论、关键统计和失败分布
8. 预览文本报告
9. 导出报告

### 5.2.3 运行约束

- 设备若处于挂载状态，GUI 与 CLI 都必须拒绝开始校验。
- 权限不足时必须返回明确错误，不允许静默失败。
- 校验取消后必须保留已完成部分的状态与结果说明。
- 报告内容必须与共享 Rust 层的格式保持一致。

## 5.3 结果表达

结果页必须同时覆盖“快速判断”和“详细分析”两层需求。

### 5.3.1 快速判断

需要直接显示：

- 最终结论
- 已校验样本数
- 设备声明容量
- 已验证容量
- 最高有效区域
- 是否中断

### 5.3.2 详细分析

需要可视化和文本化呈现：

- 24x24 样本分布图
- 读写耗时统计
- 失败类型计数
- 文本报告原文

## 5.4 报告导出

- 用户可以将报告导出为文本文件。
- GUI 应使用标准 macOS 保存面板。
- 导出失败时必须给出明确错误提示。

## 6. GUI 体验需求

## 6.1 视觉风格

整体界面应满足以下原则：

- 现代、克制、可信赖
- 优先使用 macOS 原生控件语义
- 使用清晰层级而不是重装饰
- 对风险状态保持足够显著但不过度惊吓

建议采用：

- 分栏布局
- 半透明材质卡片
- 明确的工具栏主操作
- 系统符号图标
- 统一圆角、边距与间距体系

## 6.2 动效要求

功能不多，因此动效与细节必须提升质感，但不喧宾夺主。至少应具备：

1. 设备选择切换时的平滑内容过渡
2. 校验开始、运行中、结束三种状态的显式转场
3. 进度条和关键数字变化时的柔和动画
4. Hover、按压、禁用态的细微反馈
5. 结果区块出现时的渐显或缩放过渡

动效原则：

- 以短时、低幅度、低打扰为主
- 不做连续炫技动画
- 保证 Reduce Motion 场景可降级

## 6.3 交互细节

### 6.3.1 设备列表

- 选中态明确
- Hover 有轻微高亮或抬升
- 挂载中设备不能触发校验，但仍可查看原因
- 空状态要有解释文案，不只是空白

### 6.3.2 校验过程

- 运行时主按钮切换为停止动作
- 正在运行时避免用户误触改变目标设备
- 状态文案要能区分“准备中 / 校准中 / 校验中 / 已完成 / 已取消 / 失败”
- 错误提示不能只给技术报错，要有用户可执行的建议

### 6.3.3 结果查看

- 结果摘要始终位于显眼位置
- 报告原文可复制
- 样本图每种失败状态有明确颜色/图例
- 图表与数字在深色模式和浅色模式下都保持足够对比度

## 7. CLI 体验需求

- 输出应紧凑、清晰、可复制
- 进度默认输出到 stderr，报告输出到 stdout
- 帮助文本要简洁并符合现有仓库风格
- 错误时退出码必须有意义
- 成功、失败、取消三种退出结果需要能被脚本区分

建议行为：

- 成功且无失败：退出码 0
- 校验完成但发现问题：退出码 1
- 参数错误或运行错误：退出码 2
- 用户取消：可视为失败路径，沿用非 0

## 8. 非功能需求

## 8.1 架构边界

- Rust core 继续负责校验逻辑、报告生成、共享数据结构
- macOS 设备发现和原生交互放在 macOS 专用目录
- FFI 只暴露稳定的、可序列化的能力边界

## 8.2 可靠性

- 不允许因为某个设备信息缺失导致整个列表崩溃
- 单个设备解析失败时可跳过，但要保留错误可诊断性
- GUI 不能因为后台任务而阻塞主线程
- 校验取消必须尽快响应

## 8.3 平台一致性

- 报告文本、结论语义、失败判定必须与共享 core 一致
- 样本图、计数与报告中的统计必须互相对应

## 8.4 可维护性

- macOS 共享逻辑应避免散落在 UI 视图里
- CLI 与 GUI 必须复用同一套 Swift 服务层
- 文档需清晰描述目录结构和构建流程

## 9. 错误与提示要求

系统至少需要覆盖以下错误类别：

- 没有发现设备
- 设备已挂载
- 权限不足
- 设备在校验中被移除
- FFI 返回无效数据
- 报告保存失败

每类错误都应满足：

- 有简洁的人类可读标题
- 有可执行建议
- 不吞掉底层错误

## 10. 验收标准

## 10.1 CLI

1. 可以列出 macOS 上可校验的外接整盘设备。
2. 可以对指定设备执行校验并输出进度。
3. 可以保存 Rust 生成的文本报告。
4. 在挂载设备和未确认危险操作时会拒绝执行。

## 10.2 GUI

1. 可以在启动后看到设备列表或明确空状态。
2. 可以查看所选设备的容量、路径、传输类型和状态。
3. 可以发起校验、取消校验，并看到阶段与进度变化。
4. 校验完成后可查看摘要、样本图、统计信息和文本报告。
5. 可以通过标准保存面板导出报告。
6. 切换设备、状态变化、结果出现具备细微但明确的动效反馈。

## 10.3 架构

1. GUI 与 CLI 都通过 macOS 专用 Swift 层接入 Rust FFI。
2. Rust 共享层不引入 GUI/CLI 专属逻辑。
3. README 已补充 macOS 的目录与使用方式。

## 11. 成功定义

当用户在 macOS 上首次打开项目时，能够快速理解：

- 这个项目已经具备 macOS 原生入口
- CLI 和 GUI 都能完成同一套核心校验任务
- 风险与限制被清楚说明
- 结果足够可信、足够易读、足够有系统质感
